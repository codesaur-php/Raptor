How Raptor Migrations Work

Raptor uses a file-based, forward-only migration system. Unlike traditional frameworks (e.g. Laravel), Raptor Models define their own table schema and seed data via setTable() and __initial(). Migrations are used only for schema changes after initial setup - not for creating tables.

Directory Structure
database/migrations/
├── *.sql          ← Pending (will auto-run on next request)
└── ran/
    └── *.sql      ← Executed (moved here after success)
Lifecycle
  1. Developer places a .sql file into database/migrations/
  2. MigrationMiddleware detects pending files on the next HTTP request
  3. Each SQL file is parsed and executed automatically
  4. On success, the file is moved to ran/ subfolder
  5. On failure, the -- [DOWN] section runs to clean up, and the file stays as pending for retry
SQL File Format
-- [UP]
ALTER TABLE news ADD COLUMN source VARCHAR(255) AFTER photo;
CREATE INDEX news_idx_source ON news (source);

-- [DOWN]
ALTER TABLE news DROP COLUMN source;

The -- [DOWN] section is optional. It is only used for automatic cleanup when the -- [UP] section partially fails.

Important Notes
  • Do NOT use migrations to create tables. Models auto-create tables on first use via setTable(), which calls __initial() for FK constraints and seed data
  • Forward-only: there is no rollback button. To reverse a change, write a new migration
  • File naming is free: add_source_to_news.sql, 001_fix_typo.sql - any name works
  • If a target table doesn't exist yet, the migration will fail silently and retry on subsequent requests until the Model creates the table
  • MySQL DDL statements (ALTER, CREATE INDEX) auto-commit and cannot be rolled back transactionally
