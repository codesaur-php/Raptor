How Raptor Migrations Work

Raptor uses a fully file-based migration system. State is derived from the directory layout. The database/migrations/ folder is git-ignored: each environment uploads its own SQL files through this page.

Directory layout
database/migrations/
  {userId}-{username}/
    pending_file.sql     <- pending
    ran/
      applied_file.sql   <- successfully applied
Lifecycle
  1. Click Upload SQL and pick a .sql file. It lands in your folder as a pending migration.
  2. Click the button to preview the contents and any security warnings.
  3. Click to apply. If warnings are present, type CONFIRM to proceed.
  4. On success the file moves to ran/. On failure the file stays pending; check the dashboard log for the error.
  5. To discard a typo before applying, use the button on the pending row.
SQL file format
-- Optional first-line description (used as the summary)
ALTER TABLE products ADD COLUMN category VARCHAR(100);
CREATE INDEX idx_products_category ON products (category);

Statements run in order; the first failure stops the rest and the file stays pending.

Avoid CREATE TABLE in migrations. Define a Model class with setTable() instead - Raptor auto-creates the table on the Model's first use. If you do use CREATE TABLE, the scanner will flag it and you'll need to type CONFIRM to apply.
Security guardrails

Before apply, the SQL is scanned for writes against the following sensitive tables:

  • users - password / identity
  • rbac_roles, rbac_permissions, rbac_user_role, rbac_role_permission - privilege escalation
  • organizations, organizations_users - multi-tenant boundary
  • localization_language - site-wide locale impact
  • raptor_menu - dashboard navigation

Also flagged: GRANT / REVOKE, CREATE/DROP/ALTER USER, CREATE TABLE (use a Model with setTable() instead). If any pattern matches, the Apply button requires you to type CONFIRM to proceed. This is a soft guard - a careful coder can still proceed if the migration is intentional.
