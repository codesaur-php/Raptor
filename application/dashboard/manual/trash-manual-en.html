What is Trash?

Trash is a centralized recycle bin that stores records deleted from content modules (News, Pages, Products, Reviews, Comments, References, etc.). When a content record is deleted, its full data is JSON-serialized into the trash table before being removed from the original table.

Permission Trash is accessible only to users with the system_coder role. Other admins won't see it in the sidebar.

How records end up in Trash

When you click the delete button on any content module, the following happens:

  1. Removed from main table - `deleteById($id)` removes the record from its source table
  2. Stored in Trash - the full record (before deletion) is JSON-encoded into the trash table
  3. Discord/Sidebar notification - other admins see a red badge on the "trash" sidebar item

Records deleted by other admins appear in your Trash too. Records you delete yourself also appear (unlike other modules, Trash does NOT filter out your own actions).

ActionsView / Restore / Permanently delete buttons

Actions

View record detail

Click the button on a row to open a modal showing the full deleted record as formatted JSON. Useful for inspecting content before restoring or for confirming what was deleted.

Restore a record

Click the button to open a SweetAlert confirmation. Clicking "Restore" sends the record back to its source table - first attempting the original ID, falling back to a fresh auto-increment ID if the original is taken. UNIQUE field collisions (slug, keyword, code, sku, etc.) abort the restore with a clear error. The full algorithm and the messages you may see are documented in "Restoring a record" below.

Permanently delete one record

Click the button to permanently remove a single trash record. A SweetAlert confirmation appears; click "Delete" to confirm. This is irreversible.

Empty entire Trash

The button at the top right permanently deletes all trash records at once. A confirmation dialog requires clicking "Empty Trash" before proceeding.

Warning! Empty Trash is irreversible. Use only when freeing disk space or after archiving.

Restoring a record

Click the button on a row to restore the record back into its original table. The flow:

1. UNIQUE field pre-flight check

UNIQUE columns (slug, keyword, code, sku, etc.) are detected from the schema and each is checked against the live table. If a conflict exists (for example, the deleted record's slug is already used by an active record), restore aborts with a clear error: 

Cannot restore - the following UNIQUE field(s) already have a record with the same value: slug='about-us'. Either delete the conflicting record(s) first or rename their unique value(s), then try again.

In that case, delete or rename the conflicting record, then retry restore.

2. Restore with original ID

If no UNIQUE conflict, the system tries to insert with the original ID first. On success, you'll see "Restored with original ID #X". This is the ideal case because FK references (e.g. comments.news_id) remain valid.

3. Auto-increment fallback

If the original ID is already taken (PRIMARY KEY conflict), the system retries the insert without the ID, letting the DB assign a fresh auto-increment value. A Warning appears: 

Original ID #12 is already in use. Restored with new ID #45. WARNING: Foreign key references (e.g. comments referencing this record) still point to ID #12 and need manual update.

Child records (comments, etc.) still reference the old ID and need manual SQL update: UPDATE comments SET news_id=45 WHERE news_id=12;

4. Localized content (LocalizedModel)

Multilingual modules (news, pages, products, references, texts, etc.) restore both the primary table row and all _content rows (mn, en, etc.) atomically.

5. Audit log (written to two channels)

Each restore is recorded into two log channels:

  1. trash_log with action='trash-restore' - the global Trash audit trail. Context includes restored_by, restored_at, original_id, new_id, and used_original_id.
  2. The restored module's own log channel with action='restore' and record_id=<new_id>. This entry shows up directly on the record's view/update page via the Logger Protocol. For example, a restored pages record is logged in pages_log; a restored review goes to products_log.

Controllers store the log channel name directly in the log_table column when calling TrashModel::store() (e.g. ReviewsController passes 'products', ReferencesController passes 'content'), and restore() uses that value as the log channel.

Security & permissions

  • Trash page, routes, and API all require the system_coder role.
  • Any admin with system_*_delete permission triggers a Trash entry when deleting content.
  • Records contain JSON of the original row - sensitive fields (token, password, etc.) must be masked by the source module before deletion.
  • If disk fills up, use Empty Trash to free space.
Warnings
  • Permanently deleting from Trash is irreversible.
  • Empty Trash wipes all entries at once - use carefully.
  • Don't drop the trash table manually - the UI will error trying to read from it.
