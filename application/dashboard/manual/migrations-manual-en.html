What are Migrations?

Migrations are SQL scripts that modify the existing database after initial setup - typically schema changes (ALTER TABLE, CREATE INDEX) or seed data (INSERT into translations, menus, etc.). New tables are NOT created via migrations; Model classes auto-create their tables on first use.

Permission Migrations are accessible only to users with the system_coder role.

How it works

Raptor uses a file-based migration system - state is derived from where the file lives on disk.

database/migrations/
  {userId}-{username}/
    pending_file.sql     <- pending
    ran/
      applied_file.sql   <- successfully applied

The database/migrations/ directory is git-ignored. Each environment uploads its own SQL files through this page, and the per-user folder is created automatically on first upload. This gives you a per-server audit trail of who uploaded what and which files ran successfully.

SQL file format

-- Optional first-line description (used as the summary)
ALTER TABLE products ADD COLUMN category VARCHAR(100);
CREATE INDEX idx_products_category ON products (category);
  • Statements run in order; if one fails, the rest are skipped and the file stays pending.
  • Filename convention: YYYY-MM-DD_short-description.sql for natural sorting.
  • Max file size: min(10 MB, php.ini post_max_size, php.ini upload_max_filesize) - shown in the upload button tooltip. Only .sql extension is accepted.

Lifecycle

  1. Click Upload SQL at the top of the page and pick a .sql file.
  2. The file lands in your folder as a pending migration.
  3. Click the View button to inspect the SQL, summary, SHA-256, and any security warnings.
  4. Click the Apply button. If warnings are present, the dashboard requires you to type CONFIRM (uppercase) to proceed.
  5. On success, the file moves to ran/. On failure the file stays pending; check the Logs module for the error.
  6. To discard a typo before applying, use the Delete button on the pending row.

Security guardrails

Before apply, the SQL is scanned for writes against the following sensitive tables and patterns. Any match raises a soft warning; the Apply button then requires a typed CONFIRM.

  • users - password / identity
  • rbac_roles, rbac_permissions, rbac_user_role, rbac_role_permission - privilege escalation
  • organizations, organizations_users - multi-tenant boundary
  • localization_language - site-wide locale impact
  • raptor_menu - dashboard navigation
  • GRANT / REVOKE
  • CREATE USER / DROP USER / ALTER USER
  • CREATE TABLE - prefer defining a Model with setTable(); Raptor auto-creates the table on first use

The guard is intentionally soft - a careful coder can still proceed if the migration is intentional. For routine password resets or role grants, prefer the dedicated dashboard UI over SQL migrations. For new tables, prefer Models over migrations.

Audit trail

Every upload, apply, and delete is logged to dashboard_log with full context: SHA-256 hash of the SQL, statement count, security warning count, and the destination folder. The on-disk layout itself is also an audit trail - pending files in user folders, successfully applied files in ran/.
