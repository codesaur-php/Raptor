Trash (Хогийн сав) гэж юу вэ?

Trash бол контент модулиудаас (News, Pages, Products, Reviews, Comments, References гэх мэт) устгасан бичлэгүүдийг түр хадгалах нэгдсэн төв. Контент бүхий хүснэгт устгах үйлдэл хийгдэхэд бичлэгийн бүх өгөгдлийг JSON хэлбэрээр trash хүснэгтэд хуулж аваад үндсэн хүснэгтээс устгана.

Эрх Trash хуудас нь зөвхөн system_coder дүртэй хэрэглэгч дээр нээгдэнэ. Бусад админд sidebar дээр харагдахгүй.

Бичлэг хогийн савд хэрхэн орох вэ?

Контент модул дээр устгах товч дарахад дараах үе шат тохиолдоно:

  1. Үндсэн хүснэгтээс устгана - `deleteById($id)` дуудагдаж бичлэг үндсэн хүснэгтээс хасагдана
  2. Trash-д хуулагдана - устгахаас өмнөх бичлэгийн бүх өгөгдөл trash хүснэгт рүү JSON болгож хадгалагдана
  3. Discord/Sidebar мэдэгдэл - бусад админ "trash" sidebar item дээр улаан badge харна

Бусад админ устгасан бичлэг таны trash дотор харагдана. Та өөрөө устгасан бичлэг ч мөн адил харагдана (бусад модулиас ялгаатай нь өөрийн үйлдлийг шүүхгүй).

Үйлдлүүд

Бичлэгийн дэлгэрэнгүйг харах

Хүснэгтийн мөр дээрх товчийг дарахад устгасан бичлэгийн бүх өгөгдөл JSON хэлбэрээр modal цонхонд гарч ирнэ. Энэ нь бичлэгийг сэргээхээс өмнө контентыг шалгах, эсвэл амжилттай устгасан эсэхийг батлахад тохиромжтой.

Бичлэг сэргээх (Restore)

товчийг дарахад баталгаажуулах SweetAlert цонх нээгдэнэ. "Restore" дарвал систем бичлэгийг үндсэн хүснэгтэд буцаан оруулна - анхны ID-аар оролдоод, амжилтгүй бол auto-increment ID олгоно. UNIQUE талбар (slug, keyword, code, sku) давхцалтай бол алдаа буцаана. Бүрэн алгоритм болон гарч болзошгүй мессежийн талаар "Бичлэг сэргээх" хэсгийг доороос үзнэ үү.

Нэг бичлэгийг бүрэн устгах

товчийг дарахад баталгаажуулах SweetAlert цонх гарна. "Delete" дарвал бичлэг trash хүснэгтээс үүрд устаж, сэргээх боломжгүй болно.

Бүх trash-ийг цэвэрлэх (Empty Trash)

Хуудасны баруун дээд буланд байрлах товчийг дарахад бүх trash бичлэгийг нэг дор үүрд устгана. Анхааруулах SweetAlert цонх гарах ба "Empty Trash" дарж баталгаажуулна.

Анхаар! "Empty Trash" нь сэргээх боломжгүй үйлдэл. Disk сэлбэх эсвэл архивлах шаардлагатай үед л ашиглана.

Бичлэг сэргээх (Restore)

Хүснэгтийн мөр дээрх товчийг дарахад Trash дахь бичлэг үндсэн хүснэгт рүү буцаагдана. Дараах алхмуудаар ажиллана:

1. UNIQUE талбар шалгалт (pre-flight)

Эх хүснэгтийн UNIQUE баганаудыг (slug, keyword, code, sku гэх мэт) schema-аас илрүүлж тус бүрд давхцал байгаа эсэхийг шалгана. Жишээ нь сэргээх гэж буй news бичлэгийн slug аль хэдийн өөр идэвхтэй бичлэгт ашиглагдсан бол алдаа буцаана: 

Cannot restore - the following UNIQUE field(s) already have a record with the same value: slug='about-us'. Either delete the conflicting record(s) first or rename their unique value(s), then try again.

Энэ тохиолдолд админ давхцалтай бичлэгийг устгах эсвэл нэрлэлтийг өөрчилсний дараа дахин оролдоно.

2. Анхны ID-аар сэргээх

UNIQUE асуудал байхгүй бол эхлээд анхны (original) ID-аар insert хийнэ. Амжилттай бол "Restored with original ID #X" мессеж харагдана. Энэ нь FK холбоосыг (жишээ: comments.news_id) хэвээр хадгалах учир хамгийн зөв сэргээлт юм.

3. Auto-increment fallback

Хэрэв анхны ID-аар insert хийхэд PRIMARY KEY conflict (өөр бичлэг тэр ID-г аль хэдийн эзэлсэн) гарвал систем ID-г хасч, auto-increment-аар шинэ ID олгож сэргээнэ. Энэ үед Анхааруулга гарч: 

Original ID #12 is already in use. Restored with new ID #45. WARNING: Foreign key references (e.g. comments referencing this record) still point to ID #12 and need manual update.

Хүүхэд бичлэгүүд (comments гэх мэт) нь хуучин ID руу заасан хэвээр учир тэдгээрийн FK-г шинэ ID-руу гар аргаар шинэчлэх ёстой (SQL: UPDATE comments SET news_id=45 WHERE news_id=12;).

4. Localized content (LocalizedModel)

news, pages, products, references, texts гэх мэт олон хэлтэй модул нь primary хүснэгт + _content хүснэгт хосоор сэргээгдэнэ. Олон хэлний title/content (mn, en гэх мэт) бүгд буцаж орно.

5. Audit log (хоёр тийшээ бичигдэнэ)

Сэргээх үйлдэл бүр хоёр log channel руу бичигдэнэ:

  1. trash_log - action='trash-restore'. Trash системийн ерөнхий аудит лог; restored_by, restored_at, original_id, new_id, used_original_id зэрэг бүрэн контекст хадгалагдана.
  2. Тухайн модулийн өөрийн log channel - action='restore', record_id=<new_id>. Энэ нь сэргээгдсэн бичлэгийн харах/засах хуудсан дээр Logger Protocol-ын дагуу шууд харагдана. Жишээ нь сэргээгдсэн pages бичлэг pages_log-д, reviews бичлэг products_log-д бичигдэнэ.

Trash-д бичлэг хадгалахдаа controller-ууд `log_table` баганад тухайн модулийн log channel-ийн нэрийг шууд оруулдаг (жишээ: ReviewsController нь 'products' гэж дамжуулна, ReferencesController нь 'content'). Restore хийх үед уг утгыг шууд log channel болгож ашиглана.

Аюулгүй байдал, эрх

  • Trash хуудас, route, API бүгд system_coder эрх шаарддаг.
  • Үндсэн контентыг устгах эрхтэй (system_*_delete) бүх админы үйлдэл trash-д хадгалагдана.
  • Trash хүснэгтийн өгөгдөл нь JSON хэлбэртэй учир нууц мэдээлэл (token, password гэх мэт) backend-аас урьдчилж масклагдсан байх ёстой.
  • Disk сансрын асуудал гарвал Empty Trash-ээр цэвэрлэнэ.
Анхааруулга
  • Trash дахь бичлэг устгах нь сэргээх боломжгүй.
  • Empty Trash нь зөвхөн бүх бичлэгийг нэг дор устгахад зориулагдсан, болгоомжтой ашиглана.
  • Trash хүснэгтийг manual-аар DB-аас хасахаас зайлсхийнэ - UI бичлэг харахдаа алдаа гарна.
